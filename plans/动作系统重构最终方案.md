## 目标

为当前项目建立一套更精巧、结构更强、且适合小型类鬼泣 ACT 战斗 Demo 的动作系统方案。

这套方案遵循一个核心原则：

`若无必要，勿增实体。`

因此最终只引入两个真正新的核心概念：

- `ActionRuntime`
- `ActionGraph`

其中 `ActionTransition` 只是 `ActionGraph` 中的一条数据，不作为独立系统看待。

---

## 设计结论

### 1. ActionAsset 的定位

`ActionAsset` 是动作执行的最小单位。

它只负责：

- 播什么 Timeline
- 进入/退出时附加什么 Tag
- 做什么 Cleanup
- 动作自身的基础元信息

它不再负责：

- 什么时候可以被选中
- 从哪个动作可以跳到自己
- 下一步应该跳到哪里

换句话说：

`ActionAsset` 是内容资产，不是转换规则资产。

---

### 2. Locomotion 的定位

`Locomotion` 不再被看作与 `Action` 平级竞争的一套状态机。

它只是：

`没有 Action 在播放时的默认层`

没有 Action 时，由 `ActorLocomotion` 负责：

- 默认动画
- 默认位移
- 默认旋转

一旦有 Action 在播，则默认规则是：

`Action 全权接管角色表现与控制`

也就是：

- 动画完全由 Timeline + `AnimancerTrack` 控制
- 位移与旋转默认直接使用动画 Root Motion
- 若需要覆盖默认 Root Motion，例如攻击吸附、受击特殊位移、特殊旋转，则通过 Timeline clip 实现

因此不再需要额外设计复杂的“通道实体系统”。

这里真正成立的只有一个朴素事实：

- 没有 Action 时，主导者是 `Locomotion`
- 有 Action 时，主导者是 `Action`

---

### 3. 动作切换的定位

动作切换不再由某个组件直接遍历 `ActionAsset` 列表并决定。

最终方案中：

- `ActionGraph` 负责保存所有合法转换规则
- 运行时输入、Tag、受击状态等负责提供当前事实
- `ActionRuntime` 负责评估规则并做最终裁决

因此：

`Graph 管规则，Runtime 管裁决与执行。`

---

### 4. 条件属于 Transition，不属于 Action

原有的 `ActionCondition` 体系本身是有价值的。

但更合理的挂载位置应当是：

`ActionTransition.conditions`

而不是：

`ActionAsset.EntryConditions`

因为条件的本质不是“动作本身的属性”，而是：

`从这里切到那里，需要满足什么。`

---

## 最终结构

### 保留的核心类

- `Actor`
- `ActorMovement`
- `ActionPlayer`
- `ActionAsset`
- `ActorLocomotion`
- `ActorLogicInput`
- `ActorCombater`
- `ImpactSystem`
- 现有 TimelinePlayable 全套
- 现有大部分 `ActionCondition`

### 新增的核心类

- `ActionRuntime`
- `ActionGraph`

### 删除或重写的核心类

- `ActionStateManager` -> 重写为 `ActionRuntime`

---

## 核心职责划分

### Actor

继续作为角色装配点，保存各核心组件引用。

### ActorMovement

继续作为角色唯一运动执行者。

这是底层最重要的执行组件之一，应保持：

- 统一 `CharacterController.Move()`
- 统一 Root Motion / Code Velocity / Gravity
- 统一旋转执行

### ActionPlayer

继续作为纯动作播放器，只负责：

- `Play(ActionAsset)`
- `Stop()`
- 推动 Timeline
- 上报完成/中断

它不负责决定要播哪个动作。

### ActorLocomotion

保留，但只作为默认层。

职责：

- 当没有 Action 时，驱动默认 locomotion 动画
- 当没有 Action 时，驱动默认移动与默认旋转

不再负责动作切换与状态竞争。

### ActorLogicInput

保留，继续作为输入采集层。

职责：

- 输入缓冲
- 当前按键状态
- 摇杆方向/力度

它不负责直接选 Action。

### ActorCombater

保留，负责：

- 扣血
- 死亡
- 写入待处理受击信息

它不直接播放受击动作。

### ActionGraph

只保存转换规则。

它不执行、不播放、不主动判断。

### ActionRuntime

动作运行时的核心仲裁器。

职责：

- 保存当前正在播放的 Action
- 查询 `ActionGraph`
- 评估条件
- 选择合法且优先级最高的下一个 Action
- 通知 `ActionPlayer` 播放
- 在 Action 与 Locomotion 之间切换主导权

---

## Action 与 Locomotion 的最终关系

### 状态 A：当前没有 Action

由 `ActorLocomotion` 全权负责：

- 动画
- 位移
- 旋转

### 状态 B：当前有 Action

由当前 `Action` 全权负责：

- 动画
- 位移
- 旋转

默认规则：

- 动画来自 Timeline 的 `AnimancerTrack`
- 位移和旋转默认来自 Root Motion
- 需要特殊处理时，通过 Timeline clip 覆盖

这正符合项目当前的实际工作方式。

---

## ActionGraph 设计

`ActionGraph` 可以是一个 `ScriptableObject`，内部保存：

- `List<ActionTransition> transitions`

### ActionTransition 建议字段

- `FromMode fromMode`
- `ActionAsset fromAction`
- `ActionAsset toAction`
- `List<ActionCondition> conditions`
- `int priority`

### FromMode 建议取值

- `None`
- `Current`
- `Any`

### 语义解释

#### None

表示当前没有 Action，处于默认 locomotion 状态。

例如：

- `None -> Sword_LightAttack_1`
- 条件：输入缓冲里有 `LightAttack`

#### Current

表示从当前 Action 出发。

例如：

- `Sword_LightAttack_1 -> Sword_LightAttack_2`
- 条件：输入缓冲里有 `LightAttack` 且当前存在 `ComboWindow`

#### Any

表示全局都可检查。

例如：

- `Any -> Hit_Heavy_Front`
- 条件：当前存在待处理重受击

---

## ActionRuntime 的最终工作流

### 当当前有 Action 在播时

每帧按顺序检查：

1. `Any` Transition
2. `Current` Transition

其中：

- `Any` 主要用于受击、死亡、全局强制动作
- `Current` 主要用于连段、取消、派生、恢复

如果有命中的 Transition：

- 结束当前 Action
- 播放目标 Action

如果没有命中：

- 继续播放当前 Action

如果当前 Action 自然结束：

- 若配置了 `NextAction`，则进入 `NextAction`
- 否则退出 Action，回到默认 Locomotion

### 当当前没有 Action 时

每帧按顺序检查：

1. `Any` Transition
2. `None` Transition

若无命中，则维持 Locomotion。

---

## 条件系统的最终使用方式

保留现有条件体系的大部分实现：

- `InputSequenceCondition`
- `InputStateCondition`
- `TagCondition`
- `ActionDataCondition`

但这些条件应主要用于：

`ActionTransition.conditions`

而非 `ActionAsset.EntryConditions`

这样动作条件终于回归到最合理的位置：转换规则。

---

## 受击 Action 的处理方式

### 设计原则

受击是事件驱动，但图系统是条件驱动。

两者并不冲突。

做法是：

`把事件写成一个短暂存在的运行时事实，再让 Transition 条件去读取它。`

### 最小实现方案

在 `ActorCombater` 中记录一份待处理受击数据，例如：

- `HasPendingHitReaction`
- `PendingHitReactionType`
- `PendingHitSource`

当 `TakeDamage()` 发生时：

- 先扣血
- 再解析本次命中对应的受击类型
- 写入 `PendingHitReaction`

注意：

此时不要直接播放受击动作。

### 新增一个受击条件

例如：

- `PendingHitReactionCondition`

它只负责检查：

- 当前是否存在待处理受击
- 类型是否匹配

例如图中可以写：

- `Any -> Hit_Light_Front`
- 条件：`PendingHitReaction == LightFront`

- `Any -> Hit_Heavy_Front`
- 条件：`PendingHitReaction == HeavyFront`

### 何时清除 PendingHitReaction

不要在 `TakeDamage()` 里立即清除。

应当在：

`ActionRuntime 真正接受了某个受击 Transition 并切入受击 Action 后`

再消费它。

这样可以避免丢失事件。

---

## 受击序列播放

受击动作本身仍是普通 `ActionAsset`。

其后续衔接也由图统一表达。

例如：

- `Any -> Hit_Heavy_Front`
- 条件：`PendingHitReaction == HeavyFront`

- `Hit_Heavy_Front -> Hit_Recovery`
- 条件：动作时间达到恢复窗口，或恢复 Tag 生效

- `Hit_Knockdown -> Hit_GetUp`
- 条件：起身窗口成立

因此：

受击动作不是旁路，也不是补丁，而是和普通动作完全同级的图节点。

## 这套方案解决的问题

### 1. Locomotion 与 Action 的混乱关系

现在它们不再是平级状态机。

- Locomotion 是默认层
- Action 是临时全权接管层

### 2. ActionStateManager 职责过重

原本同时承担：

- 决策
- 播放控制
- Locomotion 切换

现在收缩为 `ActionRuntime`：

- 只负责图评估、动作裁决、播放切换

### 3. ActionAsset 过重

它回归为最小动作内容单元，不再承担转换逻辑。

### 4. 受击动画难接入

通过 `PendingHitReaction + Any Transition`，受击自然进入统一图系统。

### 5. 受击后的派生与恢复难表达

现在受击动作的后续也统一由 Graph 描述。

### 6. 结构太重的问题

最终没有引入：

- Intent 系统
- Channel 实体系统
- 多套 Brain 层
- 额外复杂状态机

只引入了最必要的：

- `ActionRuntime`
- `ActionGraph`

---

## 最终一句话

这套最终方案的核心是：

`没有 Action 时，由 ActorLocomotion 作为默认层驱动角色；有 Action 时，由 Action 全权接管动画、位移与旋转；而所有动作切换都由 ActionRuntime 按 ActionGraph 中的 Transition 与当前条件进行裁决。`
