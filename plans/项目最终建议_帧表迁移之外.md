## 一、架构健康度：已经很好，但有几处"定时炸弹"

### 1. `EventCenter` — 应该淘汰或者严格限制使用范围

你的项目里实际上有**两套事件系统**：
- 旧的 `EventCenter`（基于字符串 key 的全局单例）
- 新的 `GenericEventManager<T>`（强类型，显式注册/注销）

`EventCenter` 是典型的"万能事件总线"反模式 —— 字符串 key 没有编译期检查，`static Dictionary` 不跟随场景生命周期，且没有 `RemoveEventListener` 的强制调用保证。

```58:63:Assets/Scripts/Event/EventCenter.cs
public class EventCenter
{
    private static EventCenter instance;
    public static EventCenter Instance { get => instance ?? (instance = new EventCenter()); }

    private static Dictionary<string, IEventInfo> eventDic = new Dictionary<string, IEventInfo>();
```

**建议**：搜一下项目里还有谁在用 `EventCenter`。如果只是遗留代码零星使用，尽早迁移到 `GenericEventManager<T>` 或者直接用 C# event（`ActorMovement.OnLanded`、`ActionPlayer.OnActionFinished` 用的就是这个模式，非常好）。最终删除 `EventCenter.cs`。

## 二、即将到来的 KCC 迁移 —— 现在应该注意什么

你在 `ActorMovement` 的注释里已经写了非常详细的 KCC 迁移规划。架构设计是正确的：意图层 / 通道管线 / 朝向管线 与底层移动执行是正交的。但有几个实操建议：

### a) `Actor.characterController` 字段耦合较深

```11:12:Assets/Scripts/Actor/Actor.cs
    public CharacterController characterController;
```

`ActorMovement` 通过 `actor.characterController.Move()` / `.isGrounded` / `.collisionFlags` 直接使用 CC。迁移 KCC 时，你需要把这些调用全部替换成 `KinematicCharacterMotor.MoveCharacter()` + `ICharacterController` 回调。

**建议**：迁移 KCC 时，让 `Actor` 持有的引用从 `CharacterController` 改为自定义接口或直接持有 `KinematicCharacterMotor`。不要一半新一半旧，一次性切干净。

### b) `_airborneFrameThreshold` 滤波不需要保留到 KCC

注释里已经说过，这里再强调 —— KCC 的 `GroundingStatus.IsStableOnGround` 已经处理了斜坡/台阶抖动，迁移后直接删除帧数滤波逻辑，`UpdateGroundState` 改为在 `PostGroundingUpdate` 回调里对比 `LastGroundingStatus` 和 `CurrentGroundingStatus`。

## 三、条件系统小改进建议

### `CheckEntry` 空条件列表返回 false — 可能导致误配置难以排查

```112:113:Assets/Scripts/ActionSystem/ActionAsset.cs
        if (_entryConditions == null || _entryConditions.Count == 0)
            return false;
```

空 EntryConditions → 永远无法进入。这在逻辑上是故意的（防止裸 ActionAsset 乱触发），但实际配表时很容易忘记加条件然后百思不得其解"为什么这个 Action 不触发"。

**建议**：在 `ActionAsset` 的 Custom Editor 或 `OnValidate` 里加一个 **警告**：如果 TriggerMode == Poll 且 EntryConditions 为空，显示黄色 HelpBox "此 Action 永远无法被触发（无准入条件）"。

## 四、打击感系统的一个潜在 Bug

```96:101:Assets/Scripts/Impact/ImpactSystem.cs
        if (gathered.SpeedEffect != null)
        {
            var speedEffect = new ActionSpeedEffect();
            speedEffect.Execute(impactData, gathered.SpeedEffect);
            if (speedEffect.IsActive)
                activeEffects.Add(speedEffect);
        }
```

每次命中都会 `new ActionSpeedEffect()`。如果在一次 HitStop 还没结束时连续命中多次（比如多段攻击），`activeEffects` 里会有多个 `ActionSpeedEffect` 同时在跑，它们都在 `Update` 里操作 `ActionPlayer.SetSpeed()`，最终恢复时可能会出现竞争条件（先恢复的那个把速度设回 1，后恢复的那个再设一次 1 → 问题不大；但如果 Effect 有 lerp 恢复，就可能闪烁）。

**建议**：考虑在 `ApplyImpact` 前先检查 `activeEffects` 里是否已经有活跃的同类 Effect，有的话要么覆盖（重置时长），要么跳过新的。

## 五、最应该投入时间的下一步

帧表迁移之后（或者说平行于帧表），项目中 **ROI 最高的下一件事** 不是加新功能，而是：

**做一个完整的"从 ActionAsset 配表到运行时表现"的端到端验证回路。**

### 理由
- Action System 架构已经非常成熟（Condition → CancelRule → CancelWindow → Priority 仲裁 → EventContext → MotionConfig → ImpactSystem），但这个链条**很长**
- 目前的测试大概率是手动在 Unity 里跑。如果配错一个帧窗口、漏配一个条件、优先级写反了，排查成本很高
- 不需要写 Unit Test，但可以做一个 **Editor Window**：输入"当前动作 = X，当前帧 = N，角色状态 = ..."，输出"这一帧 ASM 会产生哪些候选？最终选中谁？为什么其他候选落选？"。这相当于一个 **Action 仲裁调试器**

这个工具在开始大量配招（连招表、受击树、空中连段等）时会**指数级**节省调试时间。

## 总结

项目架构已经到了一个非常扎实的阶段。核心战斗管线的设计（ActionAsset → ASM 仲裁 → ActionPlayer 播放 → ActionInstance 生命周期 → ActorMovement 多通道位移 → ImpactSystem 打击感）逻辑链清晰，职责边界清楚。
